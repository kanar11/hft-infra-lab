How Data Flows / Przepływ danych



    

  1. Exchange sends data — The ITCH feed broadcasts a stream of binary messages about orders (who is buying, who is selling, at what price). This travels via UDP multicast (mc_sender.py). 
    Giełda wysyła dane — Kanał ITCH rozgłasza strumień wiadomości binarnych o zleceniach (kto kupuje, kto sprzedaje, w jakiej cenie). Przesyła się to poprzez multicast UDP (mc_sender.py).
    2. 


  2. Our system receives — mc_receiver_latency.py captures packets from the network. It can also use poll mode (kernel_bypass_sim.py) instead of waiting for interrupts — this is 5.6x faster. 
    Nasz system odbiera — mc_receiver_latency.py przechwytuje pakiety z sieci. Może również użyć trybu poll (kernel_bypass_sim.py) zamiast czekać na przerwania — jest to 5,6x szybsze.
    3. 


  3. Parser decodes — itch_parser.py converts raw bytes into readable data: "someone wants to buy 100 AAPL at 150.25". It does this in ~1000 ns per message. 
    Parser dekoduje — itch_parser.py konwertuje surowe bajty na czytelne dane: "ktoś chce kupić 100 AAPL po 150,25". Robi to w ~1000 ns na wiadomość.
    4. 


  4. Queue transfers — spsc_queue.cpp is a lock-free pipe between threads. The market data thread pushes data in, the trading thread pulls it out. 17.6M msg/sec, zero locks. 
    Kolejka przesyła — spsc_queue.cpp to kanał bez blokad między wątkami. Wątek danych rynkowych wstawia dane, wątek handlu je wyciąga. 17,6M wiad./s, zero blokad.
    5. 


  5. Order book matches — orderbook.cpp is the heart of the system. It holds all BIDs (buy) and ASKs (sell), sorted by price. When the highest BID >= lowest ASK, it executes a TRADE. 17.8M orders/sec. 
    Książka zleceń pasuje — orderbook.cpp to serce systemu. Przechowuje wszystkie BID'y (kupno) i ASK'i (sprzedaż), posortowane po cenie. Gdy najwyższy BID >= najniższy ASK, wykonuje TRANSAKCJĘ. 17,8M zleceń/s.
    6. 


  6. OMS checks risk — oms.py validates each order before sending: is the value too large? is the position too big? It tracks P&L (profit and loss). 
    OMS sprawdza ryzyko — oms.py weryfikuje każde zlecenie przed wysłaniem: czy wartość jest zbyt duża? czy pozycja jest zbyt duża? Śledzi P&L (zysk i stratę).
    7. 


  7. Order is sent — ouch_sender.py packs the order into binary OUCH format and sends it to the exchange. fix_parser.py handles the text-based version (FIX 4.2). 
    Zlecenie jest wysyłane — ouch_sender.py pakuje zlecenie w binarny format OUCH i wysyła je do giełdy. fix_parser.py obsługuje wersję tekstową (FIX 4.2).
    8. 




Kernel Tuning / Strojenie jądra



Hugepages — Instead of small memory pages (4KB), we use large ones (2MB). The CPU does fewer page table lookups = less latency. 
Zamiast małych stron pamięci (4KB), używamy dużych (2MB). CPU wykonuje mniej wyszukań tabeli stron = mniej opóźnień.



CPU Isolation (isolcpus=1) — CPU 1 is reserved ONLY for our trading process. No other process can run there. 
CPU 1 jest zarezerwowany WYŁĄCZNIE dla naszego procesu handlowego. Żaden inny proces nie może tam działać.



IRQ Affinity — All hardware interrupts (NIC, disk) are routed to CPU 0. CPU 1 remains completely undisturbed. 
Wszystkie przerwania sprzętowe (karta sieciowa, dysk) są kierowane do CPU 0. CPU 1 pozostaje całkowicie niezmądzony.



FIFO Scheduler — Our process has the highest priority. The kernel never preempts it to give time to other processes. 
Nasz proces ma najwyższy priorytet. Jądro nigdy go nie przydziela, aby dać czas innym procesom.



Network Tuning — 16MB network buffers, TCP timestamps disabled, swappiness=0 (nothing goes to disk). 
Bufory sieciowe 16MB, znaczniki czasu TCP wyłączone, swappiness=0 (nic nie trafia na dysk).
